# WSL 网络问题排查

> 如果你在 WSL 中使用 AxHost-Make，遇到**登录正常但发布失败**、或者**无法访问内网 AxHost 服务**的情况，请参考本文。

---

## 问题现象

在 WSL 中启动 `axhost-make serve` 后：

- 浏览器访问 `http://127.0.0.1:3820` 正常
- 登录 AxHost 账号正常（因为走的是浏览器 → `/api/axhost-proxy` → Node 服务端转发）
- 但**发布到线上**时提示 `{"code":500,"message":"fetch failed"}` 或 `{"code":502,"message":"Upload failed","status":400}`

---

## 根因分析

WSL2 使用独立的虚拟网卡，与 Windows 宿主机是**两个网络命名空间**。公司安全软件、VPN、内网 DNS 通常只作用于 Windows 宿主网络栈，导致 WSL 内部进程（Node 服务端）无法直接访问内网地址（如 `http://<axhost-ip>:<port>`）。

登录能通是因为浏览器 → Node 代理 → 远程的链路中，**Node 转发请求也是跑在 WSL 里**，所以如果 Node 本身连不上内网，登录也会受影响。但当你通过 Windows 端口转发做跳板后，登录能通说明网络链路已打通；此时如果发布仍报 `400`，往往是另一个问题——**打包格式不对**。

---

## 解决方案一：Windows 端口转发（跳板）

让 Windows 宿主机充当网络跳板，WSL 通过宿主机 IP 访问内网服务。

### 1. Windows 宿主机开启端口转发

以**管理员**身份打开 PowerShell，执行：

```powershell
netsh interface portproxy add v4tov4 listenport=18000 listenaddress=0.0.0.0 connectport=8000 connectaddress=<axhost-ip>
```

> 将 ``<axhost-ip>:<port>`` 替换为你的实际 AxHost 服务地址。

### 2. 防火墙放行

```powershell
New-NetFirewallRule -DisplayName "WSL2-AxHost-Forward" -Direction Inbound -LocalPort 18000 -Protocol TCP -Action Allow
```

### 3. WSL 中获取宿主机 IP

在 WSL 终端执行：

```bash
cat /etc/resolv.conf | grep nameserver | awk '{print $2}'
# 输出示例：172.x.x.x（宿主机在 WSL 中的虚拟网关 IP）
```

### 4. 修改 AxHost 服务地址

在 AxHost-Make 的**系统设置** → **AxHost 服务地址**中，将原来的地址：

```
http://`<axhost-ip>:<port>`
```

改为：

```
http://<宿主机IP>:18000
```

（IP 用第 3 步实际拿到的）

### 5. 验证

```bash
curl -v http://<宿主机IP>:18000
```

通了即可正常登录和发布。

### 回滚命令

如需删除端口转发：

```powershell
netsh interface portproxy delete v4tov4 listenport=18000 listenaddress=0.0.0.0
Remove-NetFirewallRule -DisplayName "WSL2-AxHost-Forward"
```

## 解决方案二：WSL2 Mirrored 网络模式（推荐，若系统支持）

Windows 11 22H2+ 支持让 WSL2 与宿主机**共享网络栈**，无需端口转发。

在 Windows 宿主机的 `%USERPROFILE%\.wslconfig` 中添加：

```ini
[wsl2]
networkingMode=mirrored
```

然后执行：

```powershell
wsl --shutdown
```

重启 WSL 后，WSL 内部可直接访问 `http://`<axhost-ip>:<port>``，无需任何额外配置。

## 常见问题：发布返回 400 / Upload failed

做完端口转发后登录正常，但发布仍报错：

```json
{"code":502,"message":"Upload failed","status":400}
```

### 原因

发布流程中，Node 服务端会先把项目文件打包成 `zip`，再上传到远程。打包命令原来用的是：

```bash
tar -acf project.zip -C cacheDir .
```

- **Windows 10+** 的 `tar` 实际是 `bsdtar`，原生支持 zip 格式，能生成合法 zip。
- **WSL/Linux** 的 `tar` 是 GNU tar，**不支持 zip**，生成的只是 `.zip` 后缀的 tar 归档。远程服务器解析失败，返回 400。

### 解决

在 WSL 中安装 `zip`：

```bash
sudo apt update && sudo apt install -y zip
```

框架代码已做兼容：检测到 `zip` 命令存在时，自动使用 `zip -r` 替代 `tar -acf`。

---

## 总结

| 问题 | 原因 | 解决 |
|---|---|---|
| WSL 中无法访问内网地址 | WSL2 独立网卡，公司安全软件不生效 | Windows 端口转发 或 WSL2 mirrored 模式 |
| 登录通但发布 400 | GNU tar 无法生成 zip，文件格式错误 | `sudo apt install zip` |
